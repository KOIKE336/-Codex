# Codexで使い回せるAGENT SKILLS 5選

『CODEXアプリの教科書』の購入特典として使いやすいように、`そのまま再利用しやすい役割テンプレ` だけを5本に絞ってまとめています。  
この特典では、いわゆる `skill` を `AGENTS.md に入れるルール + コピペ用プロンプト` の形にして、すぐ試せるようにしています。

## この特典の立ち位置

- これは「海外でバズったユースケース集」の補助特典です
- 主役は海外事例ですが、こちらは「実際に手を動かすための再利用テンプレ集」です
- いきなり全部使う必要はありません
- 最初は `1. Planning` か `2. Pull Request 事前レビュー` から試すのがおすすめです

## AGENT SKILLS は実際にどう使うのか

この特典でいう `AGENT SKILLS` は、`その役割でAIを動かしやすくするテンプレ` のことです。  
使い方は難しくなく、基本は次の3ステップです。

### 1. 使いたい役割を1つだけ選ぶ

最初から5本全部を入れなくて大丈夫です。  
たとえば

- まず計画を整理したい → `長期Goal計画立案エージェント`
- 差分を出す前に確認したい → `Pull Request 事前レビューエージェント`
- バグ調査で止まっている → `バグ再現と原因切り分けエージェント`

のように、今の作業に一番近いものを1本だけ選びます。

### 2. AGENTS.md の例をプロジェクトに入れる

各スキルにある `AGENTS.md に入れる例` は、`この案件では毎回こう動いてほしい` という常設ルールです。  
プロジェクトのフォルダ直下に `AGENTS.md` を置いて、その中へ必要な部分だけ入れます。

たとえば:

```md
## Review Agent

- 差分レビューでは、まず obvious bug / edge case / missing test を確認してください
- 指摘は「必ず直したい」「余裕があれば改善したい」に分けてください
- 結果は短く、箇条書きで返してください
```

これは毎回の土台になるルールです。  
ここには `今回だけの依頼` ではなく、`そのプロジェクトの間ずっと使いたい進め方` を書きます。

### 3. 作業するときにコピペ用プロンプトを送る

`コピペ用プロンプト` は、そのときの作業内容を伝えるためのものです。  
つまり役割分担はこうです。

- `AGENTS.md` = 毎回守ってほしい進め方
- `今回のプロンプト` = 今日やってほしい具体的な依頼

たとえばレビュー系なら:

```text
この変更差分を、Pull Request を出す前のレビューとして確認してください。

見てほしい観点:
- obvious bug
- edge case
- missing test
- security concern
```

この2つをセットで使うと、毎回細かく説明しなくても役割がぶれにくくなります。

## まずはこの使い方でOK

初心者のうちは、次の流れで十分です。

1. 使いたいスキルを1本選ぶ
2. AGENTS.md の例を必要な分だけ project に入れる
3. コピペ用プロンプトを送る
4. 返ってきた内容を見て、合わない指示だけ少し直す

大事なのは、**全部を一度に入れないこと** です。  
最初から何本も混ぜると、役割がぶつかって逆にわかりにくくなります。まずは1本だけで動かし、うまくいったらそのプロジェクト用に少しずつ育てるのが安全です。

## 採用した5本

1. 長期Goal計画立案エージェント
2. Pull Request 事前レビューエージェント
3. バグ再現と原因切り分けエージェント
4. UIチェック&改善エージェント
5. テストケース生成エージェント

## 1. 長期Goal計画立案エージェント

### 何をしてくれるか

大きめの目標を受け取って、`何から着手するか` `どこまでを1タスクにするか` `どういう順番で進めるか` を分解してくれるエージェントです。  
いきなり実装へ入らず、先に `計画を言語化する役` として使います。

### 向いている場面

- MVPをこれから作るとき
- 新機能を足す前に、タスクを整理したいとき
- /goal 実行の前に、失敗しにくい計画を作りたいとき

### AGENTS.md に入れる例

```md
## Planning Agent

- いきなり実装を始めず、まず短い実行計画を出してください
- 大きな目標は、3〜7個の小タスクに分解してください
- 各タスクについて、目的・対象ファイル・完了条件を明記してください
- 曖昧な部分があるときは、推測で大きく進めず、前提を箇条書きで整理してください
- 実装より先に「最初に触るべき1タスク」を提案してください
```

### コピペ用プロンプト

```text
この目標を、実装しやすい順番に分解してください。

目標:
- 会員向けダッシュボードに、売上グラフとフィルタ機能を追加したい

出力ルール:
- いきなりコードは書かない
- まず 3〜7 個の小タスクに分解する
- 各タスクに「目的」「触るファイル」「完了条件」を付ける
- 最初に着手すべき1タスクを理由つきで示す
- 曖昧な点があれば、先に確認事項として分ける
```

### 使い方のコツ

- 実装系のエージェントより先に使う
- `最大1時間で終わる単位` に分解させると失敗しにくい
- この出力をそのまま Issue や作業メモに転用しやすい

### 注意点

- 目標が曖昧すぎると、分解もふわっとしやすい
- 実装計画は参考であって、絶対の正解ではない

## 2. Pull Request 事前レビューエージェント

### 何をしてくれるか

変更差分を読んで、`スタイルの乱れ` ` obvious bug ` `テスト漏れ` `危ない変更` を人より先に洗い出してくれるエージェントです。  
人間レビューの前に `下ごしらえ` として使うのがポイントです。

### 向いている場面

- Pull Request を出す前
- 副業案件で、自分の変更を一度セルフチェックしたいとき
- チームレビューの往復を減らしたいとき

### AGENTS.md に入れる例

```md
## Review Agent

- 差分レビューでは、まず obvious bug / edge case / missing test を確認してください
- 人間が見るべき重い設計判断と、先に直せる軽い修正を分けてください
- 指摘は「必ず直したい」「余裕があれば改善したい」に分けてください
- 可能なら修正方針やサンプルコードも添えてください
- 結果は短く、箇条書きで返してください
```

### コピペ用プロンプト

```text
この変更差分を、Pull Request を出す前のレビューとして確認してください。

見てほしい観点:
- obvious bug
- edge case
- missing test
- security concern

出力ルール:
- 指摘は「必ず直したい」「余裕があれば改善したい」に分ける
- 可能なら修正案も付ける
- 最後に、レビュー担当へ見せる1〜2文の要約を書く
```

### 使い方のコツ

- いきなり全観点を詰め込まず、最初は3〜4観点に絞る
- 人間レビューの代わりではなく、事前整形として使う
- セキュリティだけは別枠で再確認したほうが安心

### 注意点

- 仕様そのものの正しさは、人が判断する必要がある
- 大きすぎる差分だと、指摘が雑になることがある

## 3. バグ再現と原因切り分けエージェント

### 何をしてくれるか

バグ報告やエラーログを受け取って、`再現手順` `怪しい原因` `次に見るべき場所` を整理してくれるエージェントです。  
初心者が一番つまずきやすい `どこから見ればいいかわからない` をかなり減らせます。

### 向いている場面

- ユーザーから「動かない」と言われたとき
- console error や stack trace が出ているとき
- 同じ不具合を何度も追いかけているとき

### AGENTS.md に入れる例

```md
## Debug Agent

- バグ調査では、まず再現手順を明文化してください
- ログ・画面・関連コードから、原因候補を2〜3個に絞ってください
- いきなり直す前に、「どの仮説を先に検証するか」を示してください
- 結果は「再現手順」「怪しい場所」「次の確認」の3つで返してください
- 推測が強い場合は、断定せず仮説として書いてください
```

### コピペ用プロンプト

```text
このバグ報告とログをもとに、原因を切り分けてください。

バグ報告:
- ログイン後に一覧画面が真っ白になる

ログ:
- console に "Cannot read properties of undefined" が出る

出力ルール:
- まず再現手順を書く
- 次に原因候補を2〜3個に絞る
- そのあと「最初に確認すべきファイル」と理由を書く
- いきなり修正せず、まず調査メモとしてまとめる
```

### 使い方のコツ

- `何が起きたか` だけでなく `どこで起きたか` を一緒に渡す
- スクショ、ログ、再現条件の3つがあると精度が上がりやすい
- 修正役とは分けて使うと、思考が散らばりにくい

### 注意点

- 環境依存バグは、再現できないと精度が落ちる
- 原因候補はあくまで仮説なので、人が裏を取る

## 4. UIチェック&改善エージェント

### 何をしてくれるか

画面のスクショや実装後UIを見て、`情報量が多すぎないか` `AIっぽい凡庸な見た目になっていないか` `余白や強弱が足りないか` を確認して、修正の方向を返してくれるエージェントです。

### 向いている場面

- LPや1ページサイトの仕上げ
- 個人開発アプリの見た目を整えたいとき
- `動くけどダサい` から抜けたいとき

### AGENTS.md に入れる例

```md
## UI Taste Agent

- UIを確認するときは、まず情報密度と視線誘導を見てください
- ありきたりなAIデザインになっていないかをチェックしてください
- 余白、見出しの強弱、配色、カードのメリハリを具体的に指摘してください
- 「直す順番」がわかるように、改善案は優先順位つきで返してください
- 単なる感想ではなく、修正指示に変換できる言葉で返してください
```

### コピペ用プロンプト

```text
このUIを確認して、初心者向けの特典ページとして読みやすいかレビューしてください。

見てほしい観点:
- 情報量が詰まりすぎていないか
- 見出しと本文の強弱があるか
- 余白が足りているか
- いかにもAIっぽい平均的デザインになっていないか

出力ルール:
- 「よい点」
- 「優先度が高い改善点」
- 「すぐ直せる修正案」
の3つに分けて返してください。
```

### 使い方のコツ

- 実装前より、1回作ったあとに使うほうが効果が高い
- 画像やスクショを渡せるときは一緒に見せる
- `好み` と `読みにくさ` を分けて指摘させると実務寄りになる

### 注意点

- 主観の入る領域なので、最終判断は人が持つ
- 目的が曖昧だと、評価もぼんやりしやすい

## 5. テストケース生成エージェント

### 何をしてくれるか

関数や機能の説明を受け取って、`正常系` `異常系` `境界値` のテストケースを洗い出してくれるエージェントです。  
テストを書く前の `考える手間` を大きく減らせます。

### 向いている場面

- 新機能を入れた直後
- リファクタ前に、壊したくない挙動を整理したいとき
- テストの何を書けばよいかわからないとき

### AGENTS.md に入れる例

```md
## Test Case Agent

- テストケースを作るときは、正常系・異常系・境界値を分けて考えてください
- 実装コードを書く前に、まずテスト観点を列挙してください
- 観点だけでなく、何を確認するテストかを1行で添えてください
- 重要度が高いものから順番に並べてください
- 必要なら、追加すべきテストデータ例も出してください
```

### コピペ用プロンプト

```text
この機能に対して、先にテストケースを整理してください。

対象:
- メールアドレスとパスワードでログインするフォーム

出力ルール:
- 正常系
- 異常系
- 境界値
に分けて列挙する
- 各ケースに「何を確認するテストか」を1行付ける
- 最後に、最初に書くべき3件だけ選ぶ
```

### 使い方のコツ

- テストコード生成より先に、観点整理だけさせると使いやすい
- `最初に書くべき3件` を選ばせると、着手しやすい
- 公開前チェックの下ごしらえとしても使える

### 注意点

- 生成されたテストが十分かどうかは、人が最終確認する
- 実装の意図が曖昧だと、テスト観点もズレやすい

## 配布のしかた

特典として配るなら、1本ごとに次の3点セットにすると使いやすいです。

- `何をしてくれるエージェントか` の短い説明
- `AGENTS.md に入れる例`
- `コピペ用プロンプト`

もしさらに満足感を上げるなら、最後に `おすすめの使いどころ` と `やりすぎ注意ポイント` を1行添えると、初心者でも使い始めやすくなります。
